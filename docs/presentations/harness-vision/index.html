Harnesses that
build themselves

NAVI is not only a code agent. It is a runtime for custom harnesses: skills as import UI, loops and graphs as local compilation, and feedback-driven evolution—optionally extending the engine with plugins and MCP.

01 — The gap

The world ships SKILL.md.
Reliability needs a system.

Ecosystem interchange is almost always prose. That is fine for sharing—and fragile for closed-loop automation unless the engine compiles and enforces the harness.

Approach What the model sees Who enforces the loop?
SKILL.md only Long procedural text Prompt-following luck
Skill + loop/graph files on disk Extra files if read Still the model
Skill + runtime-executed specs Goals, allowlists, gates The engine
Skill + materialize + feedback jobs Specs the agent wrote & revises Engine + human + metrics
02 — Two design axes

Loop × Graph × Engine

Loop engineering designs the cycle: plan → act → verify → reflect → improve. Graph engineering designs who runs, with which tools, and how edges fire.

Graph — structure Nodes · edges · roles · tool jail · human interrupts
↓ drives
Loop — dynamics Plan · execute · verify · reflect · budget · stop
↓ productized as
Harness pack Skills · goals · verifiers · eval · plugins / MCP
↻

Loop engineering

Exit criteria, feedback (score + text), retry policy, token/turn budget, and durable artifacts every iteration—not “run the agent again.”

◈

Graph engineering

Explore → implement → verify, fan-out subagents, hierarchical review. Shared state: session, goal, plan file, screenshots, reports.

explore → implement → verify ↺ fail implement · human gate

04 — Lifecycle

Three jobs around every harness

On import the engine materializes the pack. On use it runs. On feedback it evolves— including proposals for plugins and MCP when NAVI itself is the bottleneck.

JOB A

Materialize

Self-interview in thinking, research best loop/graph shape, survey NAVI + project capabilities, write loop/graph/verifiers. Never invent missing tools.

on skill install
RUN

Execute

Activate skill → goal budget & auto-continue → tools / subagents / workflow following the compiled graph, with hard gates at verify edges.

user task
JOB B

Harvest

Session traces + structured questions: what worked, what was slow, what capability was missing. Persist under runs/.

user feedback
JOB C

Evolve

Patch SKILL, loop, graph—and when needed, scaffold a local WASM plugin or MCP server so the next run is stronger than prompt alone.

self-improve
05 — Capability awareness

If NAVI doesn’t know itself,
it can’t use 100% of itself.

Materialization and evolution fail when the model only sees a static tool list. The system prompt / developer context needs a live map of the engine and the project.

◎

NAVI capability card

Goals on/off, browser, direct vs deferred tools, installed plugins, connected MCP, ready harnesses—hash-stable for prompt cache.

⌘

Project card

AGENTS.md plus structured stack, test/build commands, preview URLs, constraints— so verifiers match the real repo.

◇

Active harness card

Pack path, stop conditions, current graph node, tool allowlist, verifier ids— injected when the skill is active.

06 — Compliance

Don’t only hope the model follows the skill

Soft

SKILL.md + cards in context. Creative but fragile.

Medium

Host injects goal + next-node steering every turn. Better reliability.

Hard

Only current-node tools exposed; edge advances when verifier returns OK.

NAVI already has knobs for hard boundaries: skill allow/deny tools, plan mode write jail, goals auto-continue, subagent allowlists, Lua workflow. Materialize should compile SKILL.md into those knobs—not reinvent them as more prose.

07 — Upgrade ladder

Feedback upgrades more than prompts

A great harness doesn’t only rewrite its own instructions. It can extend the local engine.

1
Prompt / SKILL steps — policy and phrasing
2
Loop params — budget, max turns, verify order
3
Graph topology — split explorer / implementer / reviewer
4
Verifier recipes — bash, browser, structured reports
5
WASM plugin — local deterministic tools (e.g. a11y audit)
6
MCP server — external services the harness needs

Plugin install and MCP enable always require explicit approval. Project config cannot silently turn them on—consistent with NAVI security policy.

Skills are the ecosystem import UI; harnesses are the local compilation that NAVI materializes, runs with goals and tools, and recompiles from user feedback— optionally extending the engine through plugins and MCP.

09 — Built on today

Not a greenfield fantasy

Goals

Thread goals with auto-continue, budget, complete/blocked—loop driver.

Skills CLI

navi skill install, --skill, TUI activation—import surface.

Tools

Native code tools, browser, deferred power tools, WASM plugins, MCP client.

Plan mode

Markdown plan file + approval gate—design node with write jail.

Subagent · workflow

Nested agents and Lua orchestration—graph embryos.

SDK · headless

Embed and CI the same engine that powers the TUI.

Demo ambition

The winning story

Not “refactor auth.” A design-loop harness lives as a pack, installs via CLI, runs headless with a goal budget, fans explorer → implementer → verifier, and— after user feedback—opens a change that improves the harness itself. Loop engineering on graph engineering on the engine.
